# 安全代码审计（SAST）方法论

> 一套严格的源代码安全审计方法论 —— **只报告能证明可利用的漏洞**。
> 源自真实审计中不断自我质疑、过滤初始发现、直到只剩可证明问题的迭代过程。

---

## 核心原则

**只有能从"攻击者可控输入"完整追溯到"安全影响"的路径，才构成漏洞。**

看起来危险但**无法到达**或**无法利用**的代码模式，**不是**漏洞 —— 它们是需要排除的发现。

---

## 审计工作流（5 阶段）

```
Phase 1: 广泛扫描     → 识别所有安全相关代码模式（广撒网）
         ↓
Phase 2: 可达性验证   → 关键过滤层 ⭐（大多数误报在这里被淘汰）
         ↓
Phase 3: 缓解措施验证 → 检查框架/库是否已经中和漏洞
         ↓
Phase 4: 可利用性确认 → 构建完整攻击链
         ↓
Phase 5: 区分设计 vs 漏洞 → 最后过滤层
         ↓
      只有通过全部 5 阶段的发现才进入报告
```

---

## Phase 1: 广泛扫描

在代码库中识别所有安全相关模式，**广撒网，后面再过滤**。

**常见危险 sink**（详细速查见 `references/sink-patterns.md`）：

| 类别 | 关注模式 |
|------|---------|
| **注入** | 原始 SQL（`createNativeQuery`、字符串拼接）、`Runtime.exec`、`ProcessBuilder`、LDAP、OS 命令 |
| **XSS** | `innerHTML`、`v-html`、`document.write`、未转义的模板插值、富文本编辑器 |
| **路径遍历** | `new File(path)`、`FileInputStream`、`FileOutputStream` 带用户可控路径 |
| **SSRF** | `URL.openConnection()`、`HttpClient`、`RestTemplate` 带用户可控 URL |
| **反序列化** | `ObjectInputStream.readObject`、Jackson `enableDefaultTyping` |
| **认证绕过** | 安全过滤器中的排除路径、缺失的注解 |
| **密钥泄露** | 源码中硬编码密钥、密码、Token |
| **加密弱点** | ECB 模式、弱 key 长度、`Math.random()` 用于安全场景、MD5/SHA1 存密码 |
| **敏感数据暴露** | `System.out.println` 打印密钥、日志含密码、Cookie 含密钥 |

## Phase 2: 可达性验证 ⭐（关键过滤层）

对每个发现，验证 sink **是否真的可达**。这里淘汰最多误报。

### Step 1: 死代码检查

- 用 Grep 搜索包含漏洞的**类名、方法名、import 语句**
- 在整个代码库搜索**调用者**
- 零调用者 → **排除（死代码，无论代码看起来多危险）**

**真实案例：**
- `EntityManagerUtil` 有原始 SQL 注入 sink，但**零调用者** → 排除
- `DownloadUtils` 在 4 个方法中有路径遍历，但**无 import** → 排除
- `MarkdownViewer` 组件有 XSS，但**从未在任何页面使用** → 排除

### Step 2: 数据流追踪

- 从 HTTP 入口（Controller、请求处理器、API 端点）开始
- 追踪用户输入（请求参数、路径变量、请求体、Header）如何流动
- 验证输入**确实到达**危险 sink，且**未被清理或转换**
- 无法追踪完整路径 → **排除**

**真实案例：**
- `useMessage.tsx` 存在 `innerHTML` 注入，但错误消息来自**全局异常处理器**，其 `message` 字段只包含**枚举值和数据库数据** —— 从不包含 HTTP 请求参数。无可利用数据流 → 排除

### Step 3: 前后端对接检查

- **后端端点**：验证前端是否真的调用
- **前端函数**：验证是否被真实页面/路由调用
- 搜索端点路径、函数名、action 在整个代码库中的引用

**真实案例：**
- `/user/user/save` 端点存在批量赋值风险，但 `saveUserInfoAction` **从未被任何前端页面调用** → 从报告中排除（仅备注为"直接 API 调用"向量）

---

## Phase 3: 缓解措施验证

即使路径可达，也要检查**已存在的缓解措施**是否中和了漏洞。

### Step 1: 框架/库内置保护

- 不要假设库没有防御 —— **测试或验证其默认行为**
- 阅读库的文档或源码，确认是否有内置清理

**真实案例：**
- TinyMCE 的 `code` 插件允许原始 HTML 输入，但 TinyMCE 的**内置 HTML 清理器**会剥离 `onerror`、`onclick` 等事件属性。表面看（原始 HTML 进→ 渲染 HTML 出）像漏洞，但库本身已经缓解

### Step 2: 安全框架覆盖

- 检查安全过滤器（如 Sa-Token、Spring Security）**是否实际保护**该端点
- 验证过滤器链顺序和**排除列表** —— 端点可能被排除在认证之外

**真实案例：**
- `getPrivateKey` 端点在**三层**都被排除认证：网关 `SaReactorFilter`、服务层 `SaServletFilter`、`SaAnnotationInterceptor`。这让漏洞**更严重**，不是更轻

### Step 3: 输入验证和输出编码

- 检查 `@Valid`、`@NotBlank`、`@Size` 等注解
- 检查 sink 前的**手动清理或编码**
- 检查**参数化查询 vs 字符串拼接**

---

## Phase 4: 可利用性确认

对存活的发现，构建完整攻击链。

### 确认漏洞的 4 个要求

1. **攻击者能提供输入** —— 入口可达（最好无需认证）
2. **输入流向危险操作** —— 通过代码追踪验证（不是假设）
3. **无有效缓解** —— 通过检查 sanitization/encoding/框架保护确认
4. **具体影响可演示** —— 能描述攻击者获得什么

### 攻击链格式

```
攻击者 → [入口点] → [数据流路径] → [危险 sink] → [影响]
```

必须包含：
- 具体的 HTTP 请求（方法、路径、认证要求）
- 输入如何在代码中传播
- 危险操作对输入做什么
- 具体的安全影响

---

## Phase 5: 区分设计决策 vs 漏洞

**最终过滤层**。有些代码是**有意这样写的**。

### 自问三个问题

- 这个行为是**有文档或有意的**吗？
- 修复它会**破坏预期功能**吗？
- 这是**配置选择**还是**编码错误**？

**真实案例：**
- 锁屏设置 `pwd: undefined` —— 这是**自动锁定设计**（未设密码=立即锁），不是密码绕过
- `LoginVo` 在 Swagger 注解中含示例凭证 —— 是**文档示例**，不是运行时默认值
- Token 存为 `'true'` 字符串 —— 这是**会话标志**，不是加密 Token

## 常见误报模式（必读）

**务必熟记这些类别 —— 它们占应排除发现的绝大多数。**

| 类别 | 示例 | 为什么排除 |
|------|------|----------|
| **死代码** | `EntityManagerUtil` SQL 注入、`DownloadUtils` 路径遍历 | 整个代码库零调用者 |
| **无数据流** | `innerHTML` 与后端错误消息 | 后端异常处理器从不回显 HTTP 参数 |
| **框架缓解** | TinyMCE XSS、参数化查询 | 库/框架自己处理保护 |
| **未使用组件** | `MarkdownViewer` XSS | 组件定义但从未渲染 |
| **Swagger 示例** | `LoginVo` 默认凭证 | 文档元数据，非运行时值 |
| **前后端断层** | `saveUserInfoAction` 批量赋值 | 端点存在但前端从不调用 |
| **设计行为** | 锁屏自动锁定 | 有意行为被误判为绕过 |
| **过度推理** | Token `'true'` + Header 注释 | 单个事实正确但合并结论错误（后端用 Cookie 认证，不是 Header） |

**完整误报模式库** → `references/false-positive-patterns.md`

## 方法论检查清单（每个发现必过）

在将任何发现写入报告前，**逐项确认**：

- [ ] 危险 sink 代码确实存在且**可达**
- [ ] 我追踪了数据流**从入口到 sink**（不是假设）
- [ ] 我检查了调用者 —— 代码**不是死代码**
- [ ] 我验证了**没有框架/库缓解**中和它
- [ ] 我确认端点/函数**被实际使用**（不是孤立的）
- [ ] 我能描述**具体攻击链**（有明确步骤）
- [ ] 该行为是**漏洞**，不是**有意设计**
- [ ] 我能提供**复现步骤**（curl、PoC HTML 等）

**任何一项为"否"** → **排除该发现**，记入排除列表并说明原因。

---

## 严重程度分级

| 级别 | 条件 |
|------|------|
| **CRITICAL** | 无需认证的远程利用、完整系统接管、密钥/key 泄露 |
| **HIGH** | 需认证但影响重大、凭证泄露、权限提升 |
| **MEDIUM** | 需要特定条件、影响有限、深度防御弱点 |

## 工作原则总结

```
广撒网 → 严过滤 → 只报真
```

宁可漏报，不可误报。每一个 CRITICAL 都必须能**复现攻击**，每一个排除项都必须**写明理由**。

报告的**排除列表**和**确认漏洞列表**同样重要 —— 排除列表证明你做了完整审计，而不是只挑简单的报。
